# coreX AI MindLayer - Konzept fuer den Neuaufbau

## Kurzidee

Der **coreX AI MindLayer** (kurz **AIM**) ist ein VS-Code-Plugin, das AI-Assistenten einen stabilen, projektbezogenen Gedächtnis-Layer gibt. Ziel ist nicht, moeglichst viele Infos in eine Datei zu kippen, sondern relevante Projektwahrheiten gezielt auffindbar, validierbar und pflegbar zu machen.

AIM ist **nicht coreX-spezifisch**. coreX ist Ursprung, Referenzprojekt und Marketing-Herkunft. Das Plugin selbst muss fuer jedes beliebige Projekt funktionieren.

Der Name bleibt **coreX AI MindLayer**, aber das ausgelieferte Grundsystem ist projektneutral.

## Produktpositionierung

- **Name:** coreX AI MindLayer
- **Kurzname:** AIM
- **Lizenz-Ziel:** MIT
- **Herkunft:** entstanden aus dem coreX CMS Workflow
- **Produktversprechen:** AI-Projektkontext klein, strukturiert und nutzbar halten
- **Zielgruppe:** Entwickler, die mit AI-Assistenten an beliebigen Projekten arbeiten
- **Branding:** coreX/AIQIA sichtbar, aber das Tool bleibt generisch nutzbar
- **Homepage-Ziel:** bestehende AIM-Homepage spaeter mit coreX CMS neu bauen
- **Copyright/Attribution:** AIQIA/coreX sichtbar in README, Marketplace und LICENSE, ohne das Schema coreX-spezifisch zu machen

Marketing-Satz:

> AI MindLayer is a lightweight project memory layer for AI-assisted development - created from the coreX CMS workflow, built for any codebase.

## Zielbild

- Ein Projekt besitzt eine schlanke, projektneutrale `.ai.json` als schnellen Kompass.
- Groessere Informationen liegen in optionalen thematischen Detaildateien, z. B. `.ai.features.backend.details.json` oder `.ai.features.admin.details.json`.
- Das Plugin erkennt, validiert, visualisiert und pflegt dieses Oekosystem.
- AI-Tools, Copilot, ChatGPT/Codex und andere Agents bekommen klare, maschinenlesbare Projektkontexte.
- Der User kann Kontext bewusst freigeben, aktualisieren, aufraeumen und exportieren.
- Ein bestehendes Projekt kann per Command kontrolliert initialisiert werden, ohne dass Scanner-Funde automatisch als Wahrheit gespeichert werden.

## Reset-Ziel fuer den Neuaufbau

Die bestehende Extension soll gedanklich auf einen **zurueckgesetzten Stand** gebracht werden.

AIM startet neu mit:

- neutralem Grundschema
- wenigen Kerncommands
- validierbarem Datei-Oekosystem
- minimaler Tree View
- Kontext-Kopierfunktion
- Memory-Health-Pruefung
- Agent-Anker-Pruefung

AIM startet **nicht** mit:

- coreX-spezialisierten Defaults
- fertiger grosser Feature-Map
- Auto-Sync
- Mind Map
- Architecture Preview
- Research Manager
- Dokumentationsgenerator
- Scanner, der ungefragt Dateien vollschreibt

Der Auslieferungszustand soll leer und sauber sein: ein Grundgeruest, das fuer jedes Projekt sinnvoll befuellt werden kann.

## Leitprinzipien

- **Kontext reduzieren, nicht vermehren:** Das Plugin ist erfolgreich, wenn Agents weniger lesen muessen und trotzdem weniger Fehler machen.
- **Kompass statt Chronik:** `.ai.json` enthaelt nur langlebige Wahrheiten und rote Linien.
- **Details nur bei Bedarf:** Groessere Bereiche werden ausgelagert und ueber einen Index verbunden.
- **Maschinenlesbar, aber menschlich editierbar:** JSON bleibt klar strukturiert, lesbar und validierbar.
- **Kein Secret-Speicher:** Keine Zugangsdaten, Tokens, privaten Keys oder sensiblen Live-Daten.
- **Kein Auto-Muell:** Das Plugin darf Vorschlaege machen, aber nicht unkontrolliert Kontext aufblasen.
- **Projektneutraler Kern:** coreX ist Referenzprojekt und Herkunft, nicht harte Voraussetzung.
- **Keine Feature-Show:** Mind Map, Webviews, Auto-Sync und Scanner-Magie sind nur sinnvoll, wenn sie den Kontext kleiner und klarer machen.

## Minimal-Core-Fokus

Der Kernnutzen lautet:

> Was muss ein AI-Agent wissen, um in diesem Projekt keinen Unsinn zu machen?

Alles andere ist zweitrangig.

Der Minimal Core besteht deshalb nur aus:

- `.ai.json` erkennen, erstellen und validieren
- `.ai.features.index.json` und Detaildateien erkennen
- tote Referenzen und kaputte JSON-Dateien melden
- offensichtliche Secrets/Tokens markieren
- eine kleine Tree View als Orientierung anzeigen
- minimalen AI-Kontext kopieren
- Kontext fuer einen ausgewaehlten Feature-Bereich kopieren
- Agent-Anker fuer AGENTS.md, CLAUDE.md oder Copilot-Instructions pruefen/installieren
- Hinweise geben, wenn Dateien zu lang, doppelt oder veraltet wirken

Nicht im Minimal Core:

- Mind Map
- Architecture Preview Webview
- Auto-Sync
- Generate Docs
- Research Manager
- Docker Scanner
- Update Manager
- automatische Projektwahrheiten aus Scanner-Funden
- grosse AI-Integration, die einen bestimmten Anbieter voraussetzt

## Versioniert vs. lokal

AIM muss klar trennen, welche Dateien ins Repository gehoeren und welche lokal bleiben.

### Versioniert

```text
.ai.json
.ai.features.index.json
.ai.features.*.details.json
.ai.user-preferences.md
.ai.ignore
```

Diese Dateien beschreiben Projektwahrheiten und Team-Kontext.

### Lokal / nicht versioniert

```text

.ai-backups/
```

Diese Dateien enthalten:

- Initialisierungsstatus
- lokale Scanner-Zwischenergebnisse
- letzte UI-Auswahl

- lokale Backups

Sie gehoeren standardmaessig in `.gitignore`.

## Initializer fuer bestehende Projekte

Ein zentraler Command soll bestehende Projekte kontrolliert in ein AIM-Oekosystem ueberfuehren:

```text
AI MindLayer: Initialize From Project
```

Der Command soll:

- Projektstruktur scannen
- Tech-Stack grob erkennen
- vorhandene Dokumente finden: README, AGENTS.md, docs, composer.json, package.json, tests, config files
- Vorschlag fuer `.ai.json` erzeugen
- Vorschlag fuer `.ai.features.index.json` erzeugen
- sinnvolle Detaildateien vorschlagen
- moegliche rote Linien vorschlagen
- moegliche Tests/Commands vorschlagen
- Secret-/Token-Funde melden

Der Command soll **nicht**:

- automatisch Projektwahrheiten speichern
- Secrets uebernehmen
- lange Dokus ungeprueft kopieren
- aus Scanner-Funden harte Regeln machen
- coreX-Struktur auf fremde Projekte uebertragen

Workflow:

```text
Scan -> Vorschlaege -> User/AI Review -> Auswahl bestaetigen -> Dateien schreiben
```

## Agent-Anker: AIM darf nicht uebersehen werden

Der entscheidende Produktfaktor ist nicht nur das Datei-Oekosystem, sondern die **zuverlaessige Verankerung in den Einstiegspunkten der jeweiligen AI-Assistenten**.

Wenn ein Agent `.ai.json` und den Feature-Index nicht liest, bleibt AIM unsichtbar. Deshalb braucht AIM eine kleine, generierbare Anker-Schicht.

### Ziel

Jedes Projekt soll einen kurzen, tool-spezifischen Hinweis besitzen:

> Bevor du an diesem Projekt arbeitest, lies `.ai.json`. Folge den dortigen roten Linien und lade Detaildateien nur bei Bedarf ueber `.ai.features.index.json`.

### Mögliche Ankerdateien

Je nach Tool/Umgebung:

```text
AGENTS.md
CLAUDE.md
.github/copilot-instructions.md
.cursor/rules
.windsurfrules
GEMINI.md
README-AI.md
```

Nicht jedes Projekt braucht alle Dateien. AIM soll erkennen, welche Zielumgebung genutzt wird, und passende kurze Snippets vorschlagen.

### Neutraler Standard-Anker

Beispiel fuer `AGENTS.md` oder `CLAUDE.md`:

```md
## AI MindLayer

Dieses Projekt nutzt den coreX AI MindLayer.

Vor Projektarbeit:

1. Lies `.ai.json`.
2. Beachte besonders `red_lines`, `workflow` und `architecture`.
3. Nutze `.ai.features.index.json`, um nur bei Bedarf passende Detaildateien zu laden.
4. Schreibe keine Secrets in AI-Memory-Dateien.
5. Aktualisiere `.ai.json` nur bei langlebigen Projektwahrheiten, nicht fuer temporaere Aufgaben.
```

### Copilot-Anker

Beispiel fuer `.github/copilot-instructions.md`:

```md
This repository uses coreX AI MindLayer.

Before suggesting changes, inspect `.ai.json` for project rules, red lines, workflow and architecture boundaries. Use `.ai.features.index.json` only when feature-specific context is needed. Keep generated suggestions consistent with these files.
```

### Minimaler README-Hinweis

Falls kein tool-spezifischer Einstieg existiert:

```md
## AI Project Context

This project uses coreX AI MindLayer. AI assistants should start with `.ai.json` and load feature details via `.ai.features.index.json` only when needed.
```

### Plugin-Funktionen fuer Agent-Anker

Commands:

```text
AI MindLayer: Install Agent Anchors
AI MindLayer: Check Agent Anchors
AI MindLayer: Update Agent Anchors
AI MindLayer: Copy Agent Anchor Snippet
```

Funktionen:

- vorhandene Agent-Dateien erkennen
- fehlende Anker vorschlagen
- bestehende Inhalte nicht ueberschreiben
- AIM-Abschnitt per Marker einfuegen/aktualisieren
- veraltete Pfade erkennen
- Duplikate vermeiden
- tool-spezifische Snippets anbieten

Marker-Beispiel:

```md
<!-- AI-MINDLAYER:START -->
...
<!-- AI-MINDLAYER:END -->
```

### Sicherheits- und UX-Regeln

- Ankerdateien nie ungefragt ueberschreiben.
- Existierende Projektregeln respektieren.
- AIM-Anker kurz halten; keine komplette `.ai.json` hinein kopieren.
- Keine tool-spezifischen Behauptungen, wenn das Tool nicht erkannt wurde.
- Wenn AGENTS.md/CLAUDE.md bereits existieren, nur einen kleinen Abschnitt vorschlagen.
- Bei Konflikten Review anzeigen.

### MVP-Einordnung

Agent-Anker gehoeren in den Minimal Core, weil sie sicherstellen, dass AIM ueberhaupt genutzt wird.

Minimal Core sollte daher zusaetzlich enthalten:

- Check, ob mindestens ein Agent-Anker existiert
- Command zum Kopieren/Installieren eines kurzen Ankers
- Health-Warnung: “AIM files exist, but no known agent entrypoint references them”

## Altbestand aus der bestehenden Extension

Die bestehende Extension `aiqia.corex-ai-mindlayer` zeigt bereits viele Commands und Settings. Fuer den Neuaufbau sollten wir diese nicht ungeprueft uebernehmen, sondern in **behalten**, **umbenennen**, **spaeter bauen** und **verwerfen** sortieren.

### Rosinen aus dem alten Grundgeruest

Diese Ideen sind brauchbar und sollten in einer schlanken Form erhalten bleiben:

- **`$schema` im Root:** Jede `.ai.json` sollte optional auf ein Schema verweisen koennen.
- **Modulare Referenzen:** Die alte `$modules`-Idee ist gut, aber zu technisch benannt. Neu besser als `reference_docs`, `feature_index` oder `context_profiles` denken.
- **Load Priority:** `high`, `medium`, `low` ist als Kontext-Priorisierung sinnvoll, aber nur fuer Copy-/Context-Profile, nicht als komplexes Lazy-Loading-System im MVP.
- **VS-Code-Aktivierung:** `workspaceContains:.ai.json` ist richtig. Zusaetzlich sollte `.ai.features.index.json` die Extension aktivieren.
- **AJV/JSON Schema:** Schema-Validierung mit AJV ist ein guter technischer Kern.
- **Snippets/Templates:** JSON-Snippets und Templates sind nuetzlich, solange sie minimal bleiben.
- **`editUserPreferences`:** Als einfacher Command zum Oeffnen/Anlegen von `.ai.user-preferences.md` sinnvoll.
- **`ai-init` Idee:** Ein Initializer fuer bestehende Projekte ist zentral, aber neu als Review-basierter `Initialize From Project`, nicht als automatischer Wahrheitsschreiber.
- **Ignore-Konzept:** Eine `.ai.ignore` oder `.ai.ignore.json` kann sinnvoll sein, um Scanner-/Reference-Checks von bestimmten Pfaden fernzuhalten.
- **Schema fuer Fehler/Hinweise:** Alte `.ai.errors.json` kann als Inspiration fuer Diagnostics dienen, aber nicht als Pflichtdatei.
- **Lokaler Setup-Status:** Die alte `.ai.json.config` enthaelt eine gute Idee: Initialisierungsstatus getrennt vom Projektgedaechtnis speichern. Neu nur optional und lokal, z. B. `.ai.local.json` in `.gitignore`.
- **Meta-Block:** Autor, Publisher, Repo und Website sind als Extension-Metadaten nuetzlich, gehoeren aber nicht in jedes Zielprojekt-Memory.
- **Context-Key/Value-Liste:** Die alte `context.json` zeigt eine kompakte Wissensliste. Neu kann das als optionaler `facts`/`truths` Bereich in Detaildateien leben, nicht als freier Root-Dump.

### Was aus dem alten Grundgeruest bewusst nicht uebernommen wird

- **Absolute Redundanz:** Das widerspricht dem neuen Ziel. AIM soll Kontext reduzieren, nicht dieselben Infos ueberall doppeln.
- **Auto-Tasks mit Changelog/TODO/Docs-Sync:** Zu invasiv und fehleranfaellig fuer den Kern.
- **Composer-Plugin als Kernbestandteil:** Zu spezifisch. AIM soll zuerst VS-Code-Plugin bleiben.
- **PHP-Scanner als Pflicht:** Scanner ja, aber spaeter und nur als Vorschlagsmaschine.
- **Performance-Monitoring in `.ai.json`:** Speicher, Disk Usage und Load Time sind Plugin-Interna, keine Projektwahrheiten.
- **Context Management mit Speicherlimits in Projektdateien:** Zu viel App-Konfiguration im Projekt-Memory.
- **Tasks in `.ai.json`:** Aufgaben gehoeren nicht in den Root-Kompass. Wenn ueberhaupt, separat und optional.
- **Learning Experiences im Root:** Interessant als Historie, aber fuer Agents zu laut. Wenn noetig separat, nicht Standard.
- **Auto-Sync nach pre-commit/post-install/post-ai-update:** Zu viel Magie und Risiko.
- **Update-from-package:** Nicht Teil des MindLayer-Kerns.
- **Module-Caching/Compression als Projektformat:** Optimierung ist Plugin-Interna, nicht Teil des offenen Memory-Standards.
- **Pflichtmodule fuer errors/tasks/research/ml:** Zu breit fuer den Standard. Solche Dateien duerfen spaeter opt-in sein.

### Umbenennung alter Konzepte

| Alt | Neu |
| --- | --- |
| `$modules` | `reference_docs`, `features.index`, Detaildateien oder Kontextprofile |
| `loadPriority` | `contextPriority` fuer Copy-/Exportprofile |
| `.ai.modules/*` | `.ai.features.*.details.json` oder spaetere `.ai.context.*.json` |
| `.ai.auto-tasks.json` | Nicht Kern; hoechstens spaeter `.ai.automation.json` als opt-in |
| `.ai.errors.json` | Diagnostics/Rules im Plugin, optional spaeter `.ai.diagnostics.json` |
| `ai-init.php` | VS-Code Command `Initialize From Project` |
| `scannerAutoRun` | Kein Default; Scanner nur manuell und review-basiert |
| `.ai.json.config` | `.ai.local.json` oder Extension-Workspace-State, nicht versionierte Projektwahrheit |
| `.ai.json.ignore` | `.ai.ignore` oder `.ai.ignore.json` fuer Scanner/Validator/Context-Export |

### Code-Rosinen aus der alten VS-Code-Extension

Brauchbar als Inspiration:

- **Activation Events:** `workspaceContains:.ai.json` ist richtig; neu zusaetzlich `.ai.features.index.json`, `AGENTS.md` oder `CLAUDE.md` pruefen.
- **Basic JSON Validation:** Alte `validateAiJsonDocument()` ist simpel, aber guter Startpunkt; neu mit AJV, Diagnostics und Reference-Check erweitern.
- **FileSystemWatcher:** Watcher fuer `.ai.json` ist sinnvoll; neu aber ohne nervige Info-Popups bei jeder Aenderung.
- **Welcome Flow:** Wenn kein AIM existiert, kurzer Onboarding-Hinweis ist gut; neu dezenter Statusbar/Command statt staendiger Popup.
- **Tree/Webview-Dateioeffnung:** Die alte Idee, referenzierte Dateien anklickbar zu machen, bleibt gut; neu bevorzugt Tree View statt Webview.
- **Snippets:** Die alte JSON-Snippet-Einbindung ist brauchbar, solange Templates minimal bleiben.
- **Package-Metadaten:** `publisher`, `license`, `keywords`, Marketplace-Setup und TS/VSCE-Scripts sind als Scaffold brauchbar.

Nicht direkt uebernehmen:

- **Riesige `extension.ts`:** Neu modular aufbauen; keine 60KB Sammeldatei.
- **`cleanupAIMFiles`:** Gefaehrlich, weil es Projektdateien loescht. Wenn ueberhaupt spaeter als Review-basierter Cleanup mit Dateiliste und Undo-Hinweis.
- **`version-checker.ts`:** Viel zu gross fuer Kern; VS Code/Marketplace regelt Updates ausreichend.
- **`ecosystem.ts`:** Composer/NPM/Docker/PHP-Plugin-Setup ist Scope Creep.
- **Webview-heavy Visualisierung:** Spaeter optional, nicht Kern.
- **AutoDocs/Research Manager:** Zu weit weg vom Kontext-Hygiene-Ziel.
- **Terminal-Ausfuehrung fremder Scanner:** Nur mit explizitem User-Review und ohne Schreibzwang.

### 7. Projekt-Scanner

## Was diesmal nicht passieren darf

- Keine ueberladene Hauptdatei.
- Kein automatisches Vollschreiben mit Scanner-Funden.
- Keine hart coreX-spezifische Extension.
- Keine Secrets im Memory.
- Kein komplexer Webview-Overkill im MVP.
- Keine “AI macht alles automatisch”-Illusion.
- Keine kaputte Schema-Strenge, die echte Nutzung nervt.
- Kein Feature kommt in den Kern, nur weil es cool aussieht.
- Jedes Kernfeature muss eine Frage beantworten: Reduziert es Kontext oder verhindert es konkrete Agent-Fehler?

## Produktentscheidung

Der neue MindLayer ist **kein AI-Projektmanagementsystem**.

Er ist ein **Kontext-Hygiene-Werkzeug**:

- Projektwahrheiten klein halten
- Detailwissen sauber auslagern
- relevante Infos schnell kopierbar machen
- falsche oder riskante Agent-Aktionen verhindern
- Memory-Aufblaehung sichtbar machen

Alles, was diesen Zweck nicht direkt unterstuetzt, bleibt ausserhalb des Minimal Core.

## Umsetzungsschritte

### Phase 1 - Bestehendes Repo pruefen und entscheiden

- Bestehendes GitHub-Repo sichten
- brauchbare Teile identifizieren
- verhunzte/alte Teile verwerfen
- neuen Zielnamen und Paketnamen festlegen
- entscheiden, ob altes Repo bereinigt oder neues Repo erstellt wird

### Phase 2 - Schema und Dateiformat festlegen

- `.ai.json` Minimal- und Vollschema definieren
- Feature-Index-Schema definieren
- Details-Schema definieren
- optionale Felder vs. Pflichtfelder festlegen
- Namenskonventionen dokumentieren

### Phase 3 - VS-Code MVP bauen

- Extension-Grundgeruest
- Aktivierung bei `.ai.json` oder `.ai.features.index.json`
- Tree View
- Workspace Validation
- Secret-/Referenz-Check
- Templates
- Copy Minimal Context
- Copy Selected Area Context
- Memory Health Report

### Phase 4 - coreX als Referenzprojekt nutzen

- coreX `.ai.*` Dateien als Testfixture nehmen
- Validierung gegen echte Struktur testen
- UX mit echter Agent-Arbeit pruefen
- fehlende Felder und nervige Regeln nachjustieren

### Phase 5 - Cleanup-/Scanner-Funktionen

- Projektstruktur-Scanner als reine Vorschlagsmaschine
- Secret-Heuristik
- Vorschlaege fuer Detaildateien
- Split grosser Dateien
- Memory Health Report
- accept/reject Workflow fuer Scanner-Funde

### Phase 5b - Nur falls noetig: Visualisierung

- pruefen, ob eine Architektur-Preview echten Mehrwert liefert
- pruefen, ob eine Mind Map Kontext wirklich reduziert
- nur bauen, wenn sie nicht zum Feature-Selbstzweck wird

### Phase 6 - Dokumentation und Release

- README neu schreiben
- Screenshots/GIFs
- Beispielprojekt
- VSIX Build
- Marketplace-Metadaten
- GitHub Actions fuer Tests/Package

## Offene Fragen fuer spaeter

- Soll das Plugin nur JSON oder auch Markdown-Preference-Dateien offiziell verwalten?
- Brauchen wir `.ai.schema.json` im Projekt oder reicht Extension-internes Schema?
- Soll es Profile in `.ai.json` geben oder in einer separaten `.ai.tools.json`?
- Wie stark soll Copilot direkt integriert werden, ohne sich an einen Anbieter zu ketten?
- Soll es einen Export fuer ChatGPT/Codex geben, der automatisch nur relevante Details nimmt?
- Soll das System Team-Modus unterstuetzen, z. B. gemeinsame vs. persoenliche MindLayer-Dateien?
